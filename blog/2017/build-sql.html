Overview

Managing database versionning has always been a difficult part in projects. The main difficulty is that we have scripts that needs to be applyed once and others that needs to be reapply each time their version has been changed like packages or views.

Use cases

Use case 1: table creation/modification

In this use case we are creating a table in a first version and changing it in a second one.

file 1 - V1_1_0_20170119115500_table.sql (1)
file 2 - V1_1_1_20170120103000_table.sql (2)
file 3 - V1_1_1_20170120103500_table.sql (3)
1 First, we create a table in the 1.1.0 version
2 A column is added in the table by a developper (time of change: 10h30)
3 Later, in the same version the column is removed (time of change: 10h35)
Best approach is to let the 2 scripts apply in their creation order. So that, at anytime, reapplying the scripts in the correct order should match the correct final version of the table.

Use case 2: Coding package

Packages are code.

Usually code versionning is managed by git. So when we deliver a version of our application, the code of the package refelects this version.

Here there is no need to keep all the version of the packages. Only the wanted selection has to be applied, checked out from git.

Files Naming

A good strategy on file naming can avoid us painful packaging process.

The pattern

Two patterns can be deducted from the above scenari.

Apply once scripts

File naming is following more or less flyway convention.

Vn_n_n_YYYYMMDDHHMMSS_filename.sql

  • Vn_n_n: version name

  • YYYYMMDDHHMMSS: time stamp

  • filename: filename

V1_0_0_20170119115500_user-table.sql
The timestamp will ensure that scripts within the same version are executed in correct order.

Repeatable scripts

Some scripts like views, packages, grants, synonyms are scripts which are versionned.

We would like to compile the package script every time it’s changed.

R__package.sql
There’s no need to have a versioned name. This part is managed by versionning system like git.

Also I like to have grants and synonyms that I included in the release when needed:

R__grants.sql
R__synonyms.sql

Apply a release

All needed for the release is:

  1. Apply all the "V" sql files for the version in the time order

  2. Apply the corresponding version of the repeatable scripts like packages at the end

How to manage multiple environments?

Let’s imagine you have a local dev environment with multiple schemas on it.

You may need to apply all the scripts locally and to avoid too much changes here are simple rules to follow.

  1. Don’t prefix scripts whith schema name but instead do it in the main install script.

  2. Create grants and synonyms separately so that you apply them only in the target environment.
